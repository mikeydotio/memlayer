# Memlayer v0.3.0 — Large Response Support

## Context

MCP tool responses are currently truncated at 1,500 chars (search) and 2,000 chars (session summary) in `mcp/src/index.ts` to avoid blowing out Claude's context window. This makes large entries (like the original 0.1.0 implementation plan) unreadable. The fix: when a response exceeds a threshold, the server writes the full content to a persistent file, generates a summary + structural index, and returns a compact reference payload. The MCP client downloads the file to local temp storage and lets Claude decide which regions to read.

Additionally: rename `specs/0.1.0.txt` → `specs/prd.txt`, recreate the original 0.1.0 plan as `specs/0.1.0.md`, and update the roadmap.

## API Contract Decision

**Optional envelope field** on existing response models. When `large_response` is non-null, the full data is in the referenced file. The inline `results`/`messages` arrays are still populated (with raised truncation limits) as a degraded fallback. This is the standard REST pattern for additive, backward-compatible extensions — no new endpoints, no content negotiation, no breaking changes.

## Phase 1: Infrastructure (DB + Config + Docker Volume)

### `db/migrations/005_response_files.sql` (new)
```sql
CREATE TABLE IF NOT EXISTS response_files (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    file_path VARCHAR(512) NOT NULL,
    size_bytes BIGINT NOT NULL,
    content_type VARCHAR(32) NOT NULL DEFAULT 'text',
    summary TEXT,
    structural_index TEXT,
    source_endpoint VARCHAR(128) NOT NULL,
    source_params JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_accessed_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMPTZ
);

CREATE INDEX IF NOT EXISTS idx_response_files_lru
    ON response_files(last_accessed_at) WHERE deleted_at IS NULL;
CREATE INDEX IF NOT EXISTS idx_response_files_tombstones
    ON response_files(deleted_at) WHERE deleted_at IS NOT NULL;
```

## Phase 2: File Storage Service + Eviction

### `server/src/file_storage.py` (new)
- `store_response_file(content, source_endpoint, source_params, summary, index, content_type) → record` — writes file to `{file_storage_path}/{uuid}.txt`, inserts DB row, triggers sync eviction if hard limit exceeded
- `get_file_path(file_id) → str` — returns disk path, updates `last_accessed_at`, raises 404 if tombstoned/missing
- `get_total_file_size() → int` — `SUM(size_bytes) WHERE deleted_at IS NULL`
- `evict_lru_files(target_bytes) → int` — evicts oldest-accessed files until under target, sets `deleted_at` (tombstone), deletes files from disk

### `server/src/eviction.py` (new)
- `eviction_worker()` — async background task, checks total size each `eviction_interval_secs`, evicts at soft limit down to 80% of soft limit
- Follows the existing `embedding_worker()` pattern in `server/src/embeddings.py`

### `server/src/main.py` (modify)
- Import and launch `eviction_worker` in lifespan
- `os.makedirs(settings.file_storage_path, exist_ok=True)` on startup

## Phase 3: Indexing Module

### `server/src/indexing/__init__.py` (new)
Exports single public function:
```python
async def generate_index(content: str, mode: str = "off") -> tuple[str, str, str]:
    """Returns (summary, structural_index, detected_content_type)."""
```

### `server/src/indexing/detect.py` (new)
`detect_content_type(content) → "markdown" | "code" | "json" | "text"`
- JSON: try `json.loads()` on first 1024 bytes
- Markdown: 2+ heading patterns (`^#{1,6}\s`) in first 50 lines
- Code: 3+ function/class signatures (`def `, `fn `, `function `, `class `, `impl `, `struct `)
- Text: default fallback

### `server/src/indexing/heuristic.py` (new)
Content-type-aware indexers, each returning `(summary, structural_index)`:
- `index_markdown(content)` — heading tree with line numbers
- `index_code(content)` — function/class signatures with line numbers
- `index_json(content)` — top-level keys with types and array lengths
- `index_text(content)` — paragraph boundaries with first-sentence previews

Summary: <500 chars. Index: <2000 chars, formatted as line-range → description.

### `server/src/indexing/llm_index.py` (new)
Provider abstraction (same pattern as `server/src/embeddings.py`):
- Base class `LLMIndexProvider` with `async generate(content) → (summary, index)`
- `OpenAIIndexer` — uses OPENAI_API_KEY + INDEX_LLM_MODEL
- `AnthropicIndexer` — uses ANTHROPIC_API_KEY + INDEX_LLM_MODEL
- `OllamaIndexer` — uses OLLAMA_BASE_URL + INDEX_LLM_MODEL

### `server/src/indexing/summarize.py` (new)
Orchestrates based on `index_mode`:
- `"off"` → heuristic only
- `"hybrid"` → heuristic first, LLM fallback if index is sparse (<100 chars)
- `"llm-only"` → LLM only

After building each response:

## Phase 6: MCP Client Changes

### `mcp/src/api-client.ts` (modify)
- Add `LargeResponseRef` interface
- Add `large_response?` field to response interfaces
- Add `downloadFile(fileId: string): Promise<string>` method

### `mcp/src/file-cache.ts` (new)
- `FileCache` class — manages `MEMLAYER_CACHE_DIR` (default `~/.claude/memlayer/cache/`)
- `ensureCached(fileId, downloadFn) → localPath` — downloads if not cached
- `readLines(localPath, startLine, endLine) → string` — reads line range from cached file

### `mcp/src/index.ts` (modify)
- **Raise truncation limits** to 50,000 chars on both tools (belt-and-suspenders)
- **search_memory handler**: when `large_response` present, download file to cache, return summary + index + file_id to Claude
- **get_session_summary handler**: same pattern
- **New tool `read_memory_file`**:
  - Params: `file_id` (UUID), `start_line` (int, 1-indexed), `end_line` (int, 1-indexed inclusive)
  - Reads from local cache via `FileCache.readLines()`
  - Returns the line range as text content

## End-to-End Verification

1. `docker compose down && docker compose up -d --build` — clean start with migration
2. Ensure daemon is running and entries are ingested
3. Call `search_memory` with a query that returns a large result — verify summary + index returned, file downloaded to cache
4. Call `read_memory_file` with file_id + line range from the index — verify correct content returned
5. Call `get_session_summary` on a large session — same verification
6. Set low quotas via env vars, generate many large responses, verify eviction fires and tombstones appear
7. Restart server (`docker compose restart server`), verify file download URLs still work
8. Verify existing small responses still work inline (no `large_response` field)
